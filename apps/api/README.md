# AI Language Partner API

FastAPI backend for `ai-language-partner-mobile-shared-20260629-v1`.

This app implements the shared mobile contract in `../../contracts/openapi_v0.yaml`.
It runs without external API keys by default: LLM, TTS, and STT use mock providers, while
learner-scoped conversations, course catalog, review cards, events, XP/gamification records, TTS cache entries, and usage records persist in SQLite.

## Run

```bash
cd apps/api
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --require-hashes -r requirements.txt
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

The backend requires Python 3.11 or newer. The committed requirement files are
hash-locked; regenerate them from the corresponding `.in` source files when
updating dependencies.

Optional DB override:

```bash
AI_LANGUAGE_PARTNER_DB_PATH=/tmp/ai_language_partner.sqlite3 uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Optional local headers:

```bash
X-Learner-Id: local-dev
X-Admin-Key: local-dev-admin
```

`X-Learner-Id` scopes local/dev data. For hosted-style checks, set:

```bash
AI_LANGUAGE_PARTNER_AUTH_MODE=token
AI_LANGUAGE_PARTNER_AUTH_SECRET=...
```

Then send `X-Learner-Token` generated by `app.main.create_signed_learner_token`.
Generated tokens use expiring `v2:{learner}:{expiresAtUnix}:{hmac}` format by default.
The `local-dev-admin` audit key works only in dev mode; set `AI_LANGUAGE_PARTNER_ADMIN_KEY` outside dev.

Account auth endpoints are also implemented:

```text
POST   /v1/auth/register
POST   /v1/auth/login
POST   /v1/auth/oidc
POST   /v1/auth/oauth/pkce/start
POST   /v1/auth/oauth/pkce/callback
POST   /v1/auth/refresh
GET    /v1/auth/me
POST   /v1/auth/logout
GET    /v1/auth/devices
POST   /v1/auth/devices/attestation/challenge
POST   /v1/auth/devices/trust
DELETE /v1/auth/devices/{deviceId}
GET    /v1/auth/sessions
DELETE /v1/auth/sessions/{sessionId}
POST   /v1/auth/logout-all
POST   /v1/auth/change-password
DELETE /v1/auth/account
```

Passwords are stored as PBKDF2 hashes. Access tokens are returned as HS256 JWT bearer tokens with `accessTokenFormat: jwt_hs256`; refresh tokens remain opaque. Only token hashes are stored in SQLite. In production/hosted mode, set `AI_LANGUAGE_PARTNER_JWT_SECRET` or `AI_LANGUAGE_PARTNER_AUTH_SECRET`. Refresh rotates sessions; reusing an already-rotated refresh token returns `401`, revokes that account's sessions, and writes `auth_refresh_reuse_detected` to the audit log. Logout revokes sessions, password change revokes prior sessions, and account deletion requires password re-authentication before learner-scoped privacy deletion/account disable. Supplying `deviceId` at register/login/refresh/change-password enables optional `X-Device-Id` binding for that session and records the device as `untrusted`; `/v1/auth/devices/attestation/challenge` issues one-time challenges for configured HMAC attestation, local RS256 public-key proof-of-possession, and WebAuthn-style ES256 public-key assertion; `/v1/auth/devices/trust` can either mark the current bound device trusted by account confirmation, verify signed HMAC evidence as `signed_challenge_hmac`, verify RSA public JWK evidence as `public_key_challenge_rs256`, or verify P-256 WebAuthn assertion evidence as `webauthn_assertion_es256` with challenge, origin, RP ID hash, user-presence, authenticatorData, and clientDataJSON checks; `/v1/auth/devices/{deviceId}` revokes matching sessions and blocks future reuse of the same device id. Native Apple App Attest/Play Integrity production verification and real platform-authenticator WebAuthn ceremony evidence are still separate gaps. Login failures are throttled with `AI_LANGUAGE_PARTNER_AUTH_MAX_FAILURES` and `AI_LANGUAGE_PARTNER_AUTH_FAILURE_WINDOW_SECONDS`; account creation bursts are independently throttled with `AI_LANGUAGE_PARTNER_AUTH_REGISTER_MAX_ATTEMPTS` and `AI_LANGUAGE_PARTNER_AUTH_REGISTER_WINDOW_SECONDS`.

Pseudonymous email identifiers use keyed HMAC-SHA256. Set
`AI_LANGUAGE_PARTNER_EMAIL_HASH_SECRET` to an independent stable secret, or the
service derives the key from `AI_LANGUAGE_PARTNER_JWT_SECRET` /
`AI_LANGUAGE_PARTNER_AUTH_SECRET`. With none configured, local/test mode uses a
process-local random key and hashes are intentionally not stable across restarts.

Account session management can list active sessions, revoke a specific remote session, or revoke all sessions with optional `keepCurrent=true`.

Memory summary is implemented at `GET /v1/memory/summary`. Review cards expose HLR-inspired `memoryStrengthDays`, `memoryDifficulty`, `recallProbability`, and `recallRisk`; recommendations include `memorySummary` for tag-level recall pressure. `scripts/evaluate_learner_model.py` trains/evaluates an offline logistic learner-memory model from local review-grade rows plus deterministic fixtures and writes a reproducible JSON artifact; this is readiness evidence, not a production-trained Birdbrain replacement. `scripts/evaluate_reputation_model.py` does the same for XP abuse/reputation profiles; it is anti-cheat model readiness evidence, not production learned enforcement.

Gamification endpoints include XP ledger-backed streaks, daily quests, friend graph/invites, friend recommendations, friend quests, reward currency ledger, reward shop, reward inventory, XP boosts, achievements, league tier status, boosted-XP/duplicate-payload anomaly flags, leaderboard exclusion metadata, admin abuse review, and weekly leaderboard:

## Provider adapters

Default mode is fully local mock mode. External adapters can be enabled with env vars:

```bash
AI_LANGUAGE_PARTNER_LLM_PROVIDER=openai_compatible
AI_LANGUAGE_PARTNER_LLM_API_KEY=...
AI_LANGUAGE_PARTNER_LLM_BASE_URL=https://api.openai.com/v1
AI_LANGUAGE_PARTNER_LLM_MODEL=gpt-4o-mini
AI_LANGUAGE_PARTNER_LLM_RESPONSE_FORMAT=json_object

AI_LANGUAGE_PARTNER_TTS_PROVIDER=openai
AI_LANGUAGE_PARTNER_TTS_API_KEY=...
AI_LANGUAGE_PARTNER_TTS_MODEL=gpt-4o-mini-tts

AI_LANGUAGE_PARTNER_STT_PROVIDER=openai
AI_LANGUAGE_PARTNER_STT_API_KEY=...
AI_LANGUAGE_PARTNER_STT_MODEL=whisper-1
```

Zero-LLM dialogue bank mode adds stateless matching and prebuilt persona packs. Runtime requests never call an LLM:

```bash
cd ../..
python3 authoring/generate_seed_bank.py
python3 authoring/validate_bank.py
python3 authoring/batch_tts.py --persona yui --dry-run
./apps/api/.venv/bin/python scripts/dialogue_smoke.py
```

Optional local engines:

```bash
cd ../..
./scripts/setup_voice_engine.sh
AI_LANGUAGE_PARTNER_TTS_PROVIDER=voicevox AI_LANGUAGE_PARTNER_VOICE_ENGINE_URL=http://127.0.0.1:10101 AI_LANGUAGE_PARTNER_STT_PROVIDER=whisper_cpp apps/api/.venv/bin/uvicorn app.main:app --app-dir apps/api --host 127.0.0.1 --port 8001
launchctl kickstart -k gui/$(id -u)/com.ai-language-partner.aivis
launchctl kickstart -k gui/$(id -u)/com.ai-language-partner.whisper-server
```

If VOICEVOX/Aivis or whisper.cpp is unavailable, responses keep working through fallback providers and expose that fact in `provider` and `/v1/providers/status`. Dialogue packs are served at `/v1/dialogue/packs`, `/v1/dialogue/packs/{personaId}/{packVersion}.zip`; matching is `/v1/dialogue/match`, and missed utterances go to `/v1/dialogue/unmatched` for `authoring/weekly_expand.py`.

Check active providers:

```bash
curl http://localhost:8000/v1/providers/status
```

The LLM adapter requires `schemaVersion=turn_payload_v1` in structured turn output. Providers with strict JSON Schema support can use `AI_LANGUAGE_PARTNER_LLM_RESPONSE_FORMAT=json_schema`. Structured-output repair defaults to two attempts and is capped at three with `AI_LANGUAGE_PARTNER_LLM_REPAIR_ATTEMPTS`.

Course catalog:

```bash
curl http://localhost:8000/v1/courses
curl http://localhost:8000/v1/courses/jp_beginner_speaking_ko
curl http://localhost:8000/v1/courses/jp_travel_survival_ko
```

Current seed depth is 2 courses, 10 units, 20 lessons, and 52 practice rooms.

Content admin QA:

```bash
curl -H 'X-Admin-Key: local-dev-admin' http://localhost:8000/v1/content/quality-report
curl -X POST http://localhost:8000/v1/content/validate \
  -H 'X-Admin-Key: local-dev-admin' \
  -H 'Content-Type: application/json' \
  -d '{"courses":[],"practiceRooms":[]}'
```

`POST /v1/content/import` defaults to dry-run mode. Send `dryRun: false` to apply a validated bundle and derive room course/unit/lesson metadata from the course tree.

Content operations can be queued with `/v1/content/operations/jobs`; publisher workers can run one job with `/v1/content/operations/jobs/run-next` or run a managed scheduler tick with `/v1/content/scheduler/run-once`, which records release-worker and operation-job results in `/v1/content/scheduler/runs`. Admin HTML consoles are available at `/v1/admin/content-console` for validation/import dry-runs/bulk QA/translation memory/branch assignment actions, `/v1/admin/ops-console` for release/scheduler/job action controls, and `/v1/admin/experiment-console` for experiment analytics plus upsert/status/decision/apply action controls; all require the same `X-Admin-Key`/role headers as the admin APIs.

External provider readiness can be checked without returning secrets:

```bash
.venv/bin/python scripts/verify_docker_smoke.py
.venv/bin/python scripts/verify_external_provider_readiness.py
.venv/bin/python scripts/verify_hosted_scheduler_readiness.py
.venv/bin/python scripts/verify_redis_rate_limit_readiness.py
```

By default these record skipped checks for live services while still checking static wiring, local scheduler behavior, and secret redaction where possible. Set `AI_LANGUAGE_PARTNER_DOCKER_SMOKE_REAL_RUNS=1` only on a machine with Docker/Podman/nerdctl when you intentionally want image build, container run, and `/health` smoke evidence. Set `AI_LANGUAGE_PARTNER_EXTERNAL_SMOKE_REAL_CALLS=1` with real OIDC/OpenAI/ElevenLabs env vars, `AI_LANGUAGE_PARTNER_HOSTED_SCHEDULER_REAL_CALLS=1` with hosted scheduler health/run URLs, or `AI_LANGUAGE_PARTNER_REDIS_SMOKE_REAL_CALLS=1` with `AI_LANGUAGE_PARTNER_REDIS_URL` only when you intentionally want live provider, hosted scheduler, or Redis calls.
